LangChain & LangGraph

Complete Study Guide

PART 1: LANGCHAIN

What is LangChain?

LangChain is a Python framework that provides building blocks for LLM applications: prompt templates, chains (sequences of LLM calls), retrievers, agents, and memory. Think of it as the "glue" that connects LLMs to your application logic.

LoanIQ uses LangChain components throughout: ChatPromptTemplate, StrOutputParser, PydanticOutputParser, LCEL chains in every agent, and RecursiveCharacterTextSplitter in ingestion.

Sub-topic 1: Chains & LCEL

What, Why, How

What: A chain is a sequence of steps (LLM calls, tool invocations, data transformations). LCEL (LangChain Expression Language) is the modern way to compose chains using the pipe | operator.

Why: Instead of manually calling llm.invoke(), formatting the output, calling another function, you declare the entire pipeline declaratively. LCEL chains are automatically streamable, batchable, and async.

How:

from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

# LCEL chain — read left to right, | means "pipe output to next"
chain = prompt | llm | output_parser

# Under the hood:
# 1. prompt.invoke({"question": "..."}) → ChatPromptValue
# 2. llm.invoke(ChatPromptValue)        → AIMessage
# 3. output_parser.invoke(AIMessage)    → str
result = await chain.ainvoke({"question": "What is the DTI limit?"})

Runnable Interface

Every LCEL component implements the Runnable interface: | Method | Description | |---|---| | invoke(input) | Synchronous single call | | ainvoke(input) | Async single call | | batch([inputs]) | Synchronous list of inputs | | abatch([inputs]) | Async list of inputs | | stream(input) | Synchronous streaming | | astream(input) | Async streaming |

In LoanIQ Project

Every agent builds an LCEL chain. Example from the faithfulness evaluator:

chain = prompt | llm | StrOutputParser()
raw = await chain.ainvoke({"context": context_text, "answer": answer})

Sub-topic 2: Prompt Templates

What, Why, How

What: Prompt templates are reusable, parameterized prompt structures. Instead of building strings manually, you define a template with placeholders.

Why: Hard-coded strings are a maintenance nightmare. Templates are testable, versionable, and composable. They also handle multi-message (system/human/AI) chat formats automatically.

Types:

from langchain_core.prompts import ChatPromptTemplate, PromptTemplate, MessagesPlaceholder

# Single-string template
simple = PromptTemplate.from_template("Analyze this loan: {loan_data}")

# Multi-message chat template
chat = ChatPromptTemplate.from_messages([
    ("system", "You are a mortgage expert. Answer only from policy context."),
    ("human", "Context:\n{context}\n\nQuestion: {question}"),
])

# With conversation history
with_history = ChatPromptTemplate.from_messages([
    ("system", "You are a helpful assistant."),
    MessagesPlaceholder("chat_history"),  # inject past messages here
    ("human", "{question}"),
])

In LoanIQ Project

The RAGAS evaluator uses ChatPromptTemplate:

prompt = ChatPromptTemplate.from_messages([
    ("system", """Rate if the answer is fully supported by the context.
Return ONLY a number between 0.0 and 1.0."""),
    ("human", "CONTEXT:\n{context}\n\nANSWER:\n{answer}\n\nScore:"),
])

Sub-topic 3: Output Parsers

What, Why, How

What: Output parsers extract structured data from LLM text responses. LLMs output text; parsers convert that text into Python objects.

Types:

from langchain_core.output_parsers import StrOutputParser, JsonOutputParser
from langchain_core.output_parsers import PydanticOutputParser
from pydantic import BaseModel

# StrOutputParser — raw string
parser = StrOutputParser()  # AIMessage → str

# PydanticOutputParser — validated object
class LoanDecision(BaseModel):
    outcome: str
    reason: str
    confidence: float

pydantic_parser = PydanticOutputParser(pydantic_object=LoanDecision)
# Automatically adds format instructions to prompt
# Validates LLM output against schema
# Raises ValidationError if LLM returns wrong format

# JsonOutputParser — dict
json_parser = JsonOutputParser()

In LoanIQ Project

Every LoanIQ agent uses PydanticOutputParser to enforce structured output from the LLM. For example, the DecisionAgent parses:

class FinalDecisionOutput(BaseModel):
    outcome: Literal["APPROVE", "DECLINE", "MANUAL_REVIEW", "COUNTER_OFFER"]
    confidence_score: float = Field(ge=0, le=1)
    decision_summary: str
    decline_reasons: Optional[list[str]] = None
    conditions: Optional[list[str]] = None

If the LLM returns malformed JSON, the parser raises an exception → the agent's try/except sets a fallback verdict.

Sub-topic 4: Retrievers

What, Why, How

What: Retrievers are LangChain's interface for fetching relevant documents. They abstract over vector stores, BM25, APIs, etc.

Key retrievers:

# VectorStoreRetriever — wraps any vector store
retriever = vectorstore.as_retriever(
    search_type="similarity",
    search_kwargs={"k": 10}
)

# MultiQueryRetriever — auto-generates variants
from langchain.retrievers import MultiQueryRetriever
multi_retriever = MultiQueryRetriever.from_llm(
    retriever=base_retriever, llm=llm  # LLM generates 3 query variants
)

# EnsembleRetriever — combines dense + sparse
from langchain.retrievers import EnsembleRetriever
ensemble = EnsembleRetriever(
    retrievers=[dense_retriever, bm25_retriever],
    weights=[0.5, 0.5]  # blend scores
)

In LoanIQ Project

LoanIQ implements custom retrieval rather than using LangChain's built-in retrievers because it needs: - pgvector with raw SQL for metadata filtering - Custom RRF fusion (LangChain's EnsembleRetriever uses weighted sum, not RRF) - Custom BM25 implementation over specific fields

The pattern follows LangChain's retriever interface but with custom implementation.

Sub-topic 5: Memory

What, Why, How

What: LLMs are stateless — they don't remember previous turns. Memory components solve this by injecting conversation history into each prompt.

Types:

Memory Type How Best For
ConversationBufferMemory Stores all turns in full Short conversations (<20 turns)
ConversationBufferWindowMemory Keeps last N turns Fixed context budget
ConversationSummaryMemory Summarizes old turns Long conversations
VectorStoreRetrieverMemory Embeds history, retrieves relevant turns Very long history 
from langchain.memory import ConversationBufferMemory

memory = ConversationBufferMemory(
    return_messages=True,
    memory_key="chat_history"
)
memory.save_context(
    {"input": "What is the max LTV?"},
    {"output": "Maximum LTV is 97% for primary residence."}
)
history = memory.load_memory_variables({})["chat_history"]

In LoanIQ Project

LoanIQ doesn't use LangChain memory because it's a pipeline system (not a chatbot). State is managed by LangGraph's TypedDict state schema — more explicit and debuggable than LangChain memory for multi-agent workflows.

Sub-topic 6: Agents & Tools

What, Why, How

What: A LangChain agent is an LLM that dynamically decides which tools to call to answer a question. Unlike a chain (fixed sequence), an agent adapts.

ReAct pattern:

Thought: I need to find the DTI limit
Action: search_policy[query="DTI limit conventional loan"]
Observation: "Maximum back-end DTI is 45% for conventional loans"
Thought: I now have the answer
Final Answer: The maximum DTI is 45%

Defining tools:

from langchain_core.tools import tool
from pydantic import BaseModel

class PolicySearchInput(BaseModel):
    query: str
    loan_type: str = "CONVENTIONAL"

@tool(args_schema=PolicySearchInput)
async def search_policy(query: str, loan_type: str) -> str:
    """Search mortgage policy documents. Returns relevant policy text."""
    results = await retrieve_policy(query, loan_type=loan_type)
    return "\n\n".join(r["content"] for r in results)

AgentExecutor:

from langchain.agents import AgentExecutor, create_openai_tools_agent

agent = create_openai_tools_agent(llm, [search_policy], prompt)
executor = AgentExecutor(
    agent=agent,
    tools=[search_policy],
    max_iterations=5,    # prevent infinite loops
    verbose=True,        # see thought/action/observation
    handle_parsing_errors=True  # graceful recovery from bad LLM output
)
result = await executor.ainvoke({"input": "What is the max LTV?"})

Why LoanIQ Uses LangGraph Instead

LangChain agents are good for single-agent tasks but have limitations: - No persistent shared state across agents - Can't express conditional routing (if compliance fails → stop) - Hard to debug in production - Non-deterministic flow (LLM decides what to do next)

LoanIQ needs 7 agents with dependencies, conditional stops, and full traceability → LangGraph.

Sub-topic 7: Callbacks & Tracing

What, Why, How

What: Callbacks are hooks that fire at each step of a chain/agent execution. LangSmith is a tracing service that captures all these events.

Why: In production, you need to see what happened when a decision was made — which chunks were retrieved, what the LLM was told, what it returned, how long each step took.

PART 2: LANGGRAPH

What is LangGraph?

LangGraph is a library for building stateful, multi-actor applications with LLMs. It models your application as a directed graph where: - Nodes = agents/functions (Python async functions) - Edges = connections between agents - State = shared data structure that flows through the graph

LoanIQ's entire 7-agent mortgage decisioning pipeline is a LangGraph StateGraph.

LoanIQ LangGraph Architecture:

Sub-topic 1: State & Graph

What, Why, How

What: A StateGraph takes a TypedDict (or Pydantic model) as its state schema. Every node receives the full state and returns only the fields it modified. LangGraph merges updates automatically.

Why: Shared state is the key insight of LangGraph. Agent 1's output becomes available to Agent 2 without manual passing. The state is the "memory" that persists across the entire pipeline.

from typing import TypedDict, Optional, List
from langgraph.graph import StateGraph, START, END

class LoanState(TypedDict):
    # Input
    application_id: str
    borrower_data: dict
    loan_data: dict
    property_data: dict

    # Computed by agents (each fills their own fields)
    ltv_percentage: Optional[float]
    back_end_dti_pct: Optional[float]
    policy_verdict: Optional[str]     # "PASS" or "FAIL"
    compliance_verdict: Optional[str]
    final_decision: Optional[dict]

    # Metadata
    agent_traces: List[dict]
    errors: List[str]
    total_llm_tokens_used: int

Nodes:

async def intake_agent(state: LoanState) -> dict:
    """Validates input and performs LLM semantic review."""
    # ... validate fields, call LLM ...
    return {
        "intake_verdict": "PASS",
        "agent_traces": state["agent_traces"] + [{"agent": "intake", "status": "PASS"}]
    }
    # Only return the fields this agent changed — LangGraph merges the rest

Conditional Edges

def should_continue(state: LoanState) -> str:
    """Router: if compliance failed, stop. Otherwise continue."""
    if state.get("compliance_verdict") == "FAIL":
        return "end"   # route to END node
    return "underwriting"  # route to underwriting_agent

graph.add_conditional_edges(
    "compliance_router",
    should_continue,
    {"end": END, "underwriting": "underwriting_agent"}
)

Why LangGraph over LangChain Agents?

Feature LangChain AgentExecutor LangGraph StateGraph
Flow control LLM decides next action You define exact graph
State persistence Manual, per-chain Automatic, shared across all nodes
Conditional routing Hard to implement First-class: add_conditional_edges
Parallelism Not built-in Send() API for parallel nodes
Debuggability Opaque Inspect state at every node
Production Risky (LLM might loop) Deterministic flow

Sub-topic 2: State Management

Why Agents Return Only Changed Fields

# ❌ Wrong: return the entire state
async def ratio_agent(state: LoanState) -> LoanState:
    return {**state, "ltv": 0.8, "dti": 0.42}  # Don't do this

# ✅ Correct: return only what changed
async def ratio_agent(state: LoanState) -> dict:
    return {
        "ltv_percentage": 80.0,
        "back_end_dti_pct": 42.0,
        "estimated_monthly_payment": 1850.00,
    }
    # LangGraph merges this into the full state automatically

Why? If you return the full state, you risk overwriting changes made by a previous agent. With partial updates, each agent is isolated to its own fields.

Annotated Fields (List Appending)

By default, returning a list field replaces the previous value. To append:

from typing import Annotated
from langgraph.graph import add_messages  # or write custom reducer

class LoanState(TypedDict):
    agent_traces: Annotated[List[dict], lambda a, b: a + b]  # append, don't replace

This allows every agent to add to agent_traces without overwriting each other:

# Agent 1 returns: {"agent_traces": [{"agent": "intake", ...}]}
# Agent 2 returns: {"agent_traces": [{"agent": "ratio", ...}]}
# Final state: {"agent_traces": [{"agent": "intake"}, {"agent": "ratio"}]}

State Persistence (Checkpointers)

from langgraph.checkpoint.memory import MemorySaver
from langgraph.checkpoint.sqlite import SqliteSaver

# In-memory checkpointer (dev)
memory = MemorySaver()
app = graph.compile(checkpointer=memory)

# Persistent checkpointer (production)
saver = SqliteSaver.from_conn_string("checkpoints.db")
app = graph.compile(checkpointer=saver)

# Resume a paused pipeline using thread_id
result = await app.ainvoke(state, config={"configurable": {"thread_id": "loan-123"}})

Sub-topic 3: Graph Compilation & Execution

Building and Running a Graph

from langgraph.graph import StateGraph, START, END

# 1. Define graph
workflow = StateGraph(LoanState)

# 2. Add nodes
workflow.add_node("intake", intake_agent)
workflow.add_node("ratio", ratio_agent)
workflow.add_node("policy", policy_agent)
workflow.add_node("compliance", compliance_agent)
workflow.add_node("underwriting", underwriting_agent)
workflow.add_node("decision", decision_agent)
workflow.add_node("audit", audit_agent)

# 3. Add edges (sequential)
workflow.add_edge(START, "intake")
workflow.add_edge("intake", "ratio")
workflow.add_edge("ratio", "policy")      # policy and compliance run...
workflow.add_edge("ratio", "compliance")  # ...in parallel from ratio

# 4. Conditional edge after compliance
workflow.add_conditional_edges("compliance", should_continue,
                               {"continue": "underwriting", "end": END})
workflow.add_edge("underwriting", "decision")
workflow.add_edge("decision", "audit")
workflow.add_edge("audit", END)

# 5. Compile — validates graph, returns runnable
app = workflow.compile()

# 6. Execute
final_state = await app.ainvoke({
    "application_id": "loan-123",
    "borrower_data": {...},
    "loan_data": {...},
    "property_data": {...},
    "agent_traces": [],
    "errors": [],
    "total_llm_tokens_used": 0,
})

Streaming

# Stream state updates node by node
async for chunk in app.astream(initial_state):
    for node_name, node_output in chunk.items():
        print(f"Node {node_name} completed: {node_output.keys()}")

# Stream events (fine-grained: token-by-token LLM output)
async for event in app.astream_events(initial_state, version="v2"):
    if event["event"] == "on_chat_model_stream":
        print(event["data"]["chunk"].content, end="", flush=True)

Sub-topic 4: Multi-Agent Patterns

Sequential Pipeline (LoanIQ pattern)

Intake → Ratio → Policy → Compliance → Underwriting → Decision → Audit

Each agent specializes in one task and passes state forward. Clear separation of concerns. Easy to test individual agents.

Supervisor Pattern

An orchestrator LLM routes to specialist agents:

async def supervisor(state: State) -> dict:
    """Routes to next agent based on current state."""
    response = await llm.ainvoke("Given state, which agent should run next?")
    return {"next_agent": response.agent_name}

graph.add_conditional_edges("supervisor", lambda s: s["next_agent"],
                            {"specialist_a": "agent_a", "specialist_b": "agent_b"})

LoanIQ uses sequential not supervisor because the pipeline is deterministic — compliance always comes before underwriting, always.

Error Handling in LangGraph

async def ratio_agent(state: LoanState) -> dict:
    try:
        # ... compute ratios via LLM ...
        return {"ltv_percentage": ltv, "ratio_verdict": "PASS"}
    except TimeoutError:
        return {
            "ratio_verdict": "NEEDS_REVIEW",  # safe fallback
            "errors": state["errors"] + ["RatioAgent: LLM timeout, using rule-based fallback"]
        }
    except Exception as e:
        return {
            "ratio_verdict": "NEEDS_REVIEW",
            "errors": state["errors"] + [f"RatioAgent failed: {str(e)}"]
        }

Critical: never let an agent crash the whole pipeline. Return safe fallback values. The pipeline continues. The audit agent records the error.

10 Interview Questions — LangChain & LangGraph

Q1: What is LCEL and why is it better than manually chaining LLM calls?

A: LCEL (LangChain Expression Language) is a declarative syntax using the | pipe operator to chain LangChain components.

Manually: output = output_parser.invoke(llm.invoke(prompt.invoke(inputs)))

With LCEL: chain = prompt | llm | output_parser; output = await chain.ainvoke(inputs)

Benefits: - Streaming: LCEL chains automatically support token-by-token streaming with chain.astream() - Batching: chain.abatch([input1, input2, input3]) automatically parallelizes - Async: All LCEL chains support ainvoke() for non-blocking execution - Composability: Chains can be components in larger chains - Type safety: Input/output types are preserved through the chain

In LoanIQ, every agent builds an LCEL chain for its LLM interactions (evaluation scoring, policy analysis, decision generation).

Q2: What is the difference between StateGraph and AgentExecutor in LangChain?

A:

AgentExecutor: The LLM decides at each step which tool to call (ReAct loop). Flow is non-deterministic — the LLM might call the wrong tool, loop, or get confused. Hard to test because behavior varies with model version.

StateGraph: YOU define the exact graph — which function runs after which, what conditions route where. The LLM is called INSIDE each node but cannot change the graph structure. The flow is fully deterministic and testable.

For LoanIQ: - An AgentExecutor might skip the compliance check or run underwriting before ratio calculation - A StateGraph guarantees: intake → ratio → compliance → underwriting → decision, always - Compliance FAIL always routes to END, never to underwriting

In financial decisioning, non-deterministic agent behavior is unacceptable. LangGraph is the right choice.

Q3: Explain LangGraph state management. Why do agents return only partial state?

A: LangGraph uses a reducer pattern. When a node returns {"ltv": 0.8}, LangGraph merges this into the current state with {**current_state, "ltv": 0.8} — equivalent to dict.update().

If agents returned the full state, two agents running in parallel (policy and compliance in LoanIQ) would race to write the full state — one would overwrite the other's changes.

With partial updates: - compliance_agent returns {"compliance_verdict": "PASS", "compliance_flags": [...]} - policy_agent returns {"policy_verdict": "PASS", "retrieved_policy_chunks": [...]} - LangGraph merges both: state has all fields from both agents, no conflict

For list fields that should accumulate (like agent_traces), you use Annotated[List[dict], lambda a, b: a + b] as the reducer so each agent's trace is appended rather than replaced.

Q4: How does LoanIQ implement conditional routing? Walk through the compliance failure path.

A: After the compliance agent runs, a router function inspects the state and decides the next node:

def route_after_compliance(state: LoanState) -> str:
    if state.get("has_blocking_compliance_issues"):
        return "end"          # Hard stop — blocked by compliance
    return "underwriting"     # Continue to underwriting

This function is registered with:

graph.add_conditional_edges(
    "compliance_agent",        # source node
    route_after_compliance,    # function returns string
    {
        "end": END,            # string "end" → go to END
        "underwriting": "underwriting_agent"  # string "underwriting" → continue
    }
)

When a loan violates ECOA (Equal Credit Opportunity Act) or is in a sanctioned state, has_blocking_compliance_issues = True → the pipeline terminates immediately with a DECLINE decision from the compliance agent. The underwriting, decision, and audit agents never run.

This is a hard stop — not a soft recommendation. There's no override.

Q5: What is the "lost state" bug in LangGraph and how do you avoid it?

A: The "lost state" bug occurs when a node returns the full state object instead of just the changed fields. In parallel execution:

Time 0: State = {ltv: None, dti: None, policy: None, compliance: None}
Time 1: policy_agent and compliance_agent run in parallel

compliance_agent (returns full state):
  → {ltv: None, dti: None, policy: None, compliance: "PASS"}  # overwrites policy!

policy_agent (also returns full state):
  → {ltv: None, dti: None, policy: "PASS", compliance: None}  # overwrites compliance!

Final state: compliance=None (compliance result was lost!)

Fix: Always return only the fields your agent changed:

# compliance_agent
return {"compliance_verdict": "PASS", "compliance_flags": [...]}

# policy_agent  
return {"policy_verdict": "PASS", "retrieved_policy_chunks": [...]}
# LangGraph merges both correctly

This is the most common LangGraph bug in production. Add field-level tests: after parallel nodes complete, verify both agents' output fields are in the final state.

Q6: How does LangGraph observability work in LoanIQ?

A: LoanIQ uses three-layer observability:

  1. LangSmith tracing: When LANGCHAIN_TRACING_V2=true, every node execution is automatically traced — input state, output state diff, duration, LLM tokens used. Accessible in the LangSmith UI.

  2. agent_traces field in state: Every agent appends a structured trace to the state:

return {
    "agent_traces": [{
        "agent_name": "ratio_agent",
        "status": "PASS",
        "duration_ms": 450,
        "llm_tokens": 1200,
        "details": "DTI=42%, LTV=80%"
    }]
}
  1. Structured logging: Every agent logs key metrics using structlog — searchable in CloudWatch.

In the API response from POST /loans/{id}/decide, the pipeline_metadata field includes a complete execution report: each agent's status, duration, and token count.

Q7: What's the difference between PydanticOutputParser and JsonOutputParser? Which should you use?

A:

JsonOutputParser: Parses any valid JSON from the LLM response. Returns a Python dict. No schema validation — if the LLM returns {"outcome": 123} when you expected a string, no error.

PydanticOutputParser: Takes a Pydantic model class. Parses JSON AND validates against the schema. If outcome should be a Literal["APPROVE", "DECLINE"] and the LLM returns "MAYBE", a ValidationError is raised.

For LoanIQ's agents:

class DecisionOutput(BaseModel):
    outcome: Literal["APPROVE", "DECLINE", "MANUAL_REVIEW", "COUNTER_OFFER"]
    confidence: float = Field(ge=0.0, le=1.0, description="Confidence 0-1")
    summary: str = Field(min_length=20)

parser = PydanticOutputParser(pydantic_object=DecisionOutput)
# Adds format instructions to prompt automatically
# Validates model output — confidence=1.5 raises ValidationError

Use PydanticOutputParser when: - Schema matters (financial decisions, structured data) - You need validation - Downstream code expects typed attributes

Use JsonOutputParser when: - Schema is dynamic or unknown - You just need a dict quickly in a prototype

Q8: How do you handle LLM failures (timeouts, rate limits) in LangGraph nodes?

A: Every LoanIQ agent wraps its LLM call in a try/except and returns a safe fallback:

async def policy_agent(state: LoanState) -> dict:
    try:
        # Primary: call LLM with RAG context
        response = await llm_chain.ainvoke({"context": chunks, "loan": state})
        return {"policy_verdict": response.verdict, "policy_chunks": chunks}

    except asyncio.TimeoutError:
        # Retry once with shorter timeout
        try:
            response = await asyncio.wait_for(llm_chain.ainvoke(...), timeout=10)
            return {"policy_verdict": response.verdict}
        except:
            # Both retries failed — escalate to manual review
            return {
                "policy_verdict": "NEEDS_REVIEW",
                "errors": state["errors"] + ["PolicyAgent: LLM timeout after 2 retries"]
            }

    except RateLimitError:
        await asyncio.sleep(5)  # Brief pause
        # ... retry ...

Key principle: the pipeline must never crash. A "NEEDS_REVIEW" verdict on failure means a human loan officer reviews it. The AuditAgent records the error. The borrower gets a response, not a 500 error.

For the tenacity library (used in LoanIQ's embedding service):

from tenacity import retry, stop_after_attempt, wait_exponential

@retry(stop=stop_after_attempt(3), wait=wait_exponential(min=4, max=10))
async def embed_with_retry(texts: list[str]) -> list:
    return await embed_client.embeddings.create(model="text-embedding-3-small", input=texts)

Q9: Explain the MultiQueryRetriever. How does it improve recall?

A: MultiQueryRetriever asks an LLM to generate N alternative phrasings of the original query, runs all N queries through the retriever, and merges the results.

from langchain.retrievers import MultiQueryRetriever

retriever = MultiQueryRetriever.from_llm(
    retriever=vectorstore.as_retriever(search_kwargs={"k": 5}),
    llm=llm,
    parser_key="lines"  # LLM returns one query per line
)
# Internally generates 3 variants, retrieves 5×3=15 docs, deduplicates to unique set
docs = await retriever.ainvoke("What is the DTI limit for FHA loans?")

Why it improves recall: The vector embedding of "DTI limit FHA" might not be close to chunks that say "debt-to-income ratio maximum for government-backed loans" — different words, same concept. One variant might be: "What is the maximum debt-to-income ratio for FHA mortgage applications?" — this phrasing would embed closer to the policy chunk.

In LoanIQ, multi-query expansion is implemented custom (not using LangChain's MultiQueryRetriever) because we need control over how results are fused (we use RRF, not simple deduplication).

Q10: What is MessagesPlaceholder and when would you use it in LoanIQ?

A: MessagesPlaceholder is a special prompt template component that injects a list of BaseMessage objects (conversation history) into a specific position in the prompt template.

from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder

chat_prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a mortgage expert."),
    MessagesPlaceholder("chat_history"),  # ← inject history here
    ("human", "{question}"),
])

# Usage:
history = [
    HumanMessage(content="What is my LTV?"),
    AIMessage(content="Your LTV is 75%."),
]
prompt = chat_prompt.invoke({"chat_history": history, "question": "Is PMI required?"})

In LoanIQ: Not currently used because LoanIQ is a pipeline (not a chatbot). However, if LoanIQ added a "borrower Q&A" chatbot feature where borrowers could ask follow-up questions, MessagesPlaceholder would be used to maintain context:

"My credit score is 700, will I be approved?" → "Your score is good."
"What about my DTI?" → [needs context that we're talking about the previous loan application]

MessagesPlaceholder + ConversationBufferWindowMemory (last 10 turns) would solve this.
