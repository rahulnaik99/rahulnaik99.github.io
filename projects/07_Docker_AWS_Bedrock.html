Complete Study Guide

PART 1: DOCKER

What is Docker?

Docker packages your application and ALL its dependencies (Python version, libraries, OS-level tools) into a portable, reproducible unit called a container. The container runs identically on your laptop, a CI/CD server, and in AWS.

LoanIQ uses Docker Compose to run three services locally: FastAPI app, PostgreSQL with pgvector, and (optionally) Ollama for local LLM inference.

Sub-topic 1: Core Concepts

Image vs Container

Dockerfile                       Docker Image                    Container
(Build instructions)    →        (Frozen snapshot)      →       (Running instance)

Like a recipe           →        Like a cake mold        →       Like a baked cake

docker build -t loaniq .    docker image ls              docker run loaniq
                             → loaniq:latest
Concept Description
Image Read-only template. Built from Dockerfile. Immutable.
Container Running instance of an image. Writable layer on top of image.
Dockerfile Instructions to build an image, layer by layer.
Layer caching Each RUN, COPY instruction creates a layer. Unchanged layers reuse cache → faster builds.
Registry Where images are stored: Docker Hub (public), ECR (AWS private), GHCR (GitHub).

Layer Caching — Critical Optimization

# ❌ BAD — Cache invalidated on every code change
COPY . /app
RUN pip install -r requirements.txt   # Re-runs every time! Slow.

# ✅ GOOD — Copy requirements first (rarely changes), then code
COPY requirements.txt /app/requirements.txt
RUN pip install -r requirements.txt   # Only re-runs when requirements.txt changes
COPY . /app                           # Copy code after (often changes)

Why: Docker caches each layer. If a layer changes, all subsequent layers are re-built. requirements.txt changes less often than your code — put it earlier to maximize cache hits.

Sub-topic 2: Dockerfile Best Practices

LoanIQ Dockerfile

From docker/Dockerfile:

# Multi-stage build — builder stage
FROM python:3.11-slim as builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt --target=/app/packages

# Production stage — only what's needed
FROM python:3.11-slim
WORKDIR /app
COPY --from=builder /app/packages /app/packages
COPY . /app

# Non-root user for security
RUN useradd --create-home --shell /bin/bash appuser
USER appuser

ENV PYTHONPATH=/app/packages
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]

Best Practices

Practice Why
Multi-stage builds Builder stage installs dev tools; production stage is clean. Final image has no compiler, no test deps. Can shrink image from 2GB to 200MB.
python:3.11-slim Slim variant is Debian minimal — no extras. "slim" is 50% smaller than base.
Non-root user If container is compromised, attacker has user-level access, not root.
.dockerignore Exclude __pycache__, .env, .git, *.pyc — reduces build context size, prevents secrets in image.
ENV vs ARG ARG is build-time only (not in final image). ENV persists into running container. Never put secrets in ENV (visible in docker inspect).
ENTRYPOINT vs CMD ENTRYPOINT is fixed: ["uvicorn"]. CMD is the default argument, overridable: ["app.main:app", "--port", "8000"]. Override: docker run loaniq app.main:app --port 9000.

Sub-topic 3: Docker Compose

LoanIQ docker-compose.yml

version: "3.9"

services:
  postgres:
    image: pgvector/pgvector:pg16      # Official pgvector image
    environment:
      POSTGRES_DB: loaniq_db
      POSTGRES_USER: loaniq
      POSTGRES_PASSWORD: ${DB_PASSWORD}  # from .env file
    volumes:
      - postgres_data:/var/lib/postgresql/data  # persist beyond container lifetime
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U loaniq"]
      interval: 5s
      timeout: 5s
      retries: 5

  api:
    build: .                           # build from Dockerfile in current dir
    ports:
      - "8000:8000"                    # host:container port mapping
    environment:
      DATABASE_URL: postgresql+asyncpg://loaniq:${DB_PASSWORD}@postgres:5432/loaniq_db
    env_file:
      - .env                           # load all env vars from .env
    depends_on:
      postgres:
        condition: service_healthy     # wait for postgres to be ready
    volumes:
      - ./policies:/app/policies       # mount policy PDFs

volumes:
  postgres_data:                       # named volume — persists data

Key Docker Compose Concepts

depends_on with condition: service_healthy: Simply specifying depends_on: postgres only waits for the container to start — PostgreSQL might not be ready to accept connections yet. The condition: service_healthy waits for the healthcheck to pass.

healthcheck: Defines how Docker knows a service is ready. pg_isready -U loaniq asks PostgreSQL if it's accepting connections. Docker retries every 5 seconds, up to 5 times.

Networks: Services in the same Compose file can communicate by service name. api connects to postgres using hostname postgres — Docker's internal DNS resolves this.

Volumes: - Named volume (postgres_data): Managed by Docker, persists between docker compose down calls - Bind mount (./policies:/app/policies): Maps host directory to container directory — changes on host are immediately visible in container

Wait-for-Postgres Pattern

Application startup often fails because the API container starts before PostgreSQL is ready. LoanIQ handles this with Docker healthchecks and application-level retry:

# In app/main.py startup
@app.on_event("startup")
async def startup():
    # Retry DB connection with exponential backoff
    for attempt in range(5):
        try:
            await check_db_connection()
            await create_tables()
            break
        except Exception:
            if attempt == 4:
                raise
            await asyncio.sleep(2 ** attempt)  # 1, 2, 4, 8, 16 seconds

# FastAPI health endpoint (hit by Docker healthcheck and load balancers)

PART 2: AWS BEDROCK

What is AWS Bedrock?

AWS Bedrock is Amazon's fully managed generative AI service. It provides API access to foundation models from Anthropic (Claude), Meta (Llama), Mistral, Cohere, and Amazon's own Titan models — all within your AWS account.

Key difference from OpenAI: Data never leaves your AWS account. No data sharing, no model training on your data (unless you opt in). Critical for mortgage/banking compliance.

LoanIQ uses Bedrock for PII-sensitive agents (Compliance, Audit) where data cannot leave the enterprise network.

Why Bedrock for PII Agents?

The Compliance Agent handles ECOA checks (Equal Credit Opportunity Act) which may involve processing protected class indicators (race, gender, age — for negative action analysis). The Audit Agent writes the full decision record with borrower PII.

Sending this to OpenAI or any external API violates: - GLBA (Gramm-Leach-Bliley Act) — financial data privacy - Potentially GDPR if borrowers are in EU - Internal data governance policies

With Bedrock: - Data stays in your AWS VPC - VPC endpoints: traffic never touches the public internet - CloudWatch logs every call for audit trail - Bedrock Guardrails can auto-redact PII from responses

Sub-topic 7: Invoking Models

boto3 Bedrock Client

import boto3
import json

# Method 1: invoke_model (model-specific request format)
bedrock = boto3.client("bedrock-runtime", region_name="us-east-1")

response = bedrock.invoke_model(
    modelId="anthropic.claude-haiku-4-5-20251001",
    body=json.dumps({
        "anthropic_version": "bedrock-2023-05-31",
        "max_tokens": 1000,
        "messages": [{"role": "user", "content": "Analyze this loan..."}]
    }),
    contentType="application/json",
)
result = json.loads(response["body"].read())
answer = result["content"][0]["text"]

Converse API (Unified Interface)

# Method 2: converse API — same interface for ALL Bedrock models
response = bedrock.converse(
    modelId="anthropic.claude-haiku-4-5-20251001",
    messages=[{"role": "user", "content": [{"text": "Analyze this loan..."}]}],
    system=[{"text": "You are a compliance expert."}],
    inferenceConfig={"maxTokens": 1000, "temperature": 0.1},
)
answer = response["output"]["message"]["content"][0]["text"]

The converse API works identically for Claude, Llama, Titan, and Mistral — just change modelId. No need to learn model-specific request formats.

LangChain Bedrock Wrapper

from langchain_aws import ChatBedrock

llm = ChatBedrock(
    model_id="anthropic.claude-haiku-4-5-20251001",
    model_kwargs={"temperature": 0.1, "max_tokens": 1000},
    region_name="us-east-1",
)

# Works exactly like ChatOpenAI — same LCEL chains
chain = prompt | llm | StrOutputParser()
result = await chain.ainvoke({"question": "..."})

LoanIQ's llm_router.py switches between OpenAI and Bedrock models using this interface, with the same LCEL chains — just a different llm object.

IAM Role vs Access Keys: - IAM Role: Preferred for AWS services (ECS, Lambda, EC2). Auto-rotated credentials. boto3 automatically uses the role's credentials. - Access Keys: For local development and CI/CD. Store in ~/.aws/credentials or env vars. Rotate regularly.

Model access must be enabled manually: In the Bedrock console, you must explicitly request access to each model (especially Anthropic models — requires accepting their usage policy). One-time setup per AWS account.

10 Interview Questions — Docker & AWS Bedrock

Q1: Why does LoanIQ use Docker? What problem does it solve?

A: Docker solves the classic "works on my machine" problem by packaging the application, its dependencies, and runtime configuration into a portable container.

Concrete LoanIQ problems Docker solves: 1. pgvector dependency: The pgvector PostgreSQL extension must match the exact PostgreSQL version. Without Docker, every developer installs PostgreSQL differently → version mismatches, broken vector operations. With Docker: pgvector/pgvector:pg16 gives everyone identical PostgreSQL 16 + pgvector.

  1. Python environment: LoanIQ requires Python 3.11, langchain 0.2.x, sqlalchemy 2.x, sentence-transformers. On a dev's machine with Python 3.9 and older packages, imports fail. Docker locks everything.

  2. Production parity: What runs on a developer's MacBook (ARM) also runs on AWS ECS (x86 Linux) — same Docker image. No "I can't reproduce the production bug."

  3. Onboarding: New team member → docker compose up → working system in 5 minutes.

Q2: Explain Docker layer caching. How does LoanIQ's Dockerfile maximize cache efficiency?

A: Each instruction in a Dockerfile creates a layer. Docker caches layers and reuses them if the instruction and all preceding layers are unchanged. If a layer changes, all subsequent layers are invalidated.

LoanIQ's optimization:

# Layer 1: Base image (rarely changes)
FROM python:3.11-slim

# Layer 2: System packages (occasionally changes)
RUN apt-get install -y libpq-dev

# Layer 3: Python requirements (changes when you add a package)
COPY requirements.txt .
RUN pip install -r requirements.txt  # cached until requirements.txt changes

# Layer 4: Application code (changes frequently)
COPY . /app

If you're working on app/agents/policy_agent.py and rebuild: - Layers 1-3 are served from cache (instant) - Only layer 4 is re-executed (a few seconds)

If requirements.txt had been copied after the code, every code change would invalidate the pip install layer → 3-minute reinstall on every build.

Q3: What does depends_on: condition: service_healthy do and why is it better than plain depends_on?

A:

depends_on: postgres (plain): Docker starts the API container after the PostgreSQL container starts. But "started" means the container process began — PostgreSQL might still be initializing (writing WAL, setting up extensions, accepting connections in ~2-5 seconds).

Result: API container starts, tries to connect to PostgreSQL, gets "connection refused," crashes.

depends_on: condition: service_healthy: Docker waits until the PostgreSQL container's healthcheck reports healthy — meaning PostgreSQL is actually accepting connections.

healthcheck:
  test: ["CMD-SHELL", "pg_isready -U loaniq"]  # actually tests connection
  interval: 5s
  timeout: 5s
  retries: 5

pg_isready connects to PostgreSQL and verifies it's accepting queries. Docker won't start the API container until this passes. This eliminates the race condition.

LoanIQ also has application-level retry (5 attempts with backoff) as defense-in-depth against any remaining timing issues.

Q4: Why does LoanIQ use Bedrock for the Compliance and Audit agents specifically?

A: The Compliance agent performs HMDA (Home Mortgage Disclosure Act) and ECOA (Equal Credit Opportunity Act) checks. During this process, it may process: - Borrower demographic information (age, marital status — ECOA protected classes) - Property state and location data - Full borrower financial profile

The Audit agent writes the complete decision record including: - Full borrower PII (name, SSN last 4, income, address) - Decline reasons (which could be linked to protected characteristics) - Agent reasoning traces with sensitive financial details

If these were sent to OpenAI: 1. GLBA violation risk: Financial institutions must protect customer financial information 2. Data residency: Data in OpenAI's infrastructure, potentially outside required jurisdictions 3. Audit trail: No guarantee of how data is logged or retained by OpenAI

With Bedrock (us-east-1): - Data stays in the company's AWS account, in a specific region - VPC endpoint ensures traffic never hits the public internet - CloudWatch logs every call — complete audit trail for regulators - Bedrock Guardrails can auto-redact PII from responses

Q5: How does LoanIQ route different agents to different LLM providers? Show the architecture.

A: LoanIQ uses an llm_router.py module that selects the LLM based on the agent name and environment:

# llm_router.py
MODEL_ROUTING = {
    "IntakeAgent":        {"dev": "openai/gpt-4o-mini",          "prod": "bedrock/claude-haiku"},
    "RatioAgent":         {"dev": "ollama/loaniq-ratio",          "prod": "ollama/loaniq-ratio"},
    "PolicyAgent":        {"dev": "openai/gpt-4o",               "prod": "openai/gpt-4o"},
    "ComplianceAgent":    {"dev": "bedrock/claude-haiku",         "prod": "bedrock/claude-haiku"},
    "UnderwritingAgent":  {"dev": "openai/gpt-4o",               "prod": "bedrock/claude-sonnet"},
    "DecisionAgent":      {"dev": "openai/gpt-4o",               "prod": "bedrock/claude-sonnet"},
    "AuditAgent":         {"dev": "bedrock/claude-haiku",         "prod": "bedrock/claude-haiku"},
}

def get_llm(agent_name: str) -> BaseChatModel:
    env = "prod" if settings.is_production else "dev"
    model_id = MODEL_ROUTING[agent_name][env]

    if model_id.startswith("bedrock/"):
        return ChatBedrock(model_id=BEDROCK_MODEL_IDS[model_id.split("/")[1]])
    elif model_id.startswith("openai/"):
        return ChatOpenAI(model=model_id.split("/")[1])
    elif model_id.startswith("ollama/"):
        return ChatOpenAI(base_url="http://localhost:11434/v1", model=model_id.split("/")[1])

All agents use the same LCEL chain pattern — only the llm object differs. This is the router pattern for cost optimization.

Q6: What is the Bedrock converse API and why is it preferred over invoke_model?

A:

invoke_model: Model-specific request body. Each provider has different JSON schema: - Anthropic: {"anthropic_version": ..., "messages": [...], "max_tokens": ...} - Llama: {"prompt": "<|begin_of_text|>...", "max_gen_len": ...} - Titan: {"inputText": ..., "textGenerationConfig": {...}}

Switching models requires changing the request format everywhere.

converse API: Unified interface for all models:

response = bedrock.converse(
    modelId="any-bedrock-model-id",
    messages=[{"role": "user", "content": [{"text": "..."}]}],
    system=[{"text": "..."}],
    inferenceConfig={"maxTokens": 1000, "temperature": 0.1},
)

Same code works for Claude, Llama, Mistral, Titan. Just change modelId. AWS translates to each model's native format.

LoanIQ uses ChatBedrock from LangChain which wraps the converse API — even more abstraction, compatible with LCEL chains.

Q7: How do Docker volumes work in LoanIQ? What would happen if you ran docker compose down -v?

A: LoanIQ has two volume types:

Named volume (postgres_data):

volumes:
  postgres_data:/var/lib/postgresql/data

Managed by Docker. When docker compose down (without -v), containers stop/remove but the volume persists. PostgreSQL data (all loan applications, borrower records, vector embeddings) survives restarts.

Bind mount (./policies:/app/policies):

volumes:
  - ./policies:/app/policies

Maps a host directory to the container. Changes in ./policies on your laptop immediately appear in the container. Used so you can add new policy PDFs without rebuilding the image.

docker compose down -v: Destroys ALL named volumes. In LoanIQ, this deletes the PostgreSQL database. All loan data, embeddings, and ingested policies are permanently lost. You'd need to: 1. Run python scripts/init_db.py to recreate tables 2. Run python scripts/ingest_policies.py to re-ingest all policy PDFs

In production: NEVER run docker compose down -v. Use database backups and docker compose stop → start instead.

A: Minimum required permissions:

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": ["bedrock:InvokeModel", "bedrock:InvokeModelWithResponseStream"],
    "Resource": [
      "arn:aws:bedrock:us-east-1::foundation-model/anthropic.claude-haiku-*",

Secure setup for LoanIQ:

Development (local Docker): - Create IAM user with above policy - Store AWS_ACCESS_KEY_ID and AWS_SECRET_ACCESS_KEY in .env file - .env is in .dockerignore — never in the image

Production (ECS/Fargate): - Create IAM Role for the ECS task (not user — no long-lived credentials) - Attach the policy to the role - ECS task automatically receives temporary credentials via metadata endpoint - boto3 automatically discovers these credentials

Why roles over access keys in production: IAM roles auto-rotate credentials every 15 minutes. If an access key leaks, it's valid indefinitely. If a role's temporary credential leaks, it's expired in 15 minutes.

Additionally, enable bedrock:InvokeModel only for specific model ARNs (not *) — principle of least privilege.

Q9: Compare Bedrock Knowledge Bases to LoanIQ's custom RAG. When would you choose each?

A:

Bedrock Knowledge Bases (fully managed): - Upload documents to S3 → Bedrock handles chunking, embedding, storage, retrieval - One-click setup, no infrastructure management - Basic semantic search (cosine similarity) - Limited metadata filtering - Black box — can't inspect or tune the retrieval - Cost: pay per query + storage

When to choose Bedrock KB: Prototyping, small teams without ML engineering, general-purpose document Q&A where basic semantic search is sufficient.

LoanIQ Custom RAG: - Full control over every stage: PyMuPDF extraction, recursive chunking, pgvector storage, BM25 + dense hybrid retrieval, RRF fusion, MMR diversity, cross-encoder reranking - State/loan_type metadata pre-filtering - Can tune each component independently - Requires engineering expertise to build and maintain - Cost: infrastructure (PostgreSQL, compute)

Why LoanIQ chose custom: 1. Hybrid retrieval for exact mortgage code/number matching (BM25 is critical) 2. State-level metadata filtering (only return California-relevant policies for a California borrower) 3. Cross-encoder reranking for precision (compliance errors are high-stakes) 4. Full observability: know exactly which chunks were retrieved and why for every decision

For a mortgage compliance system, the default retrieval accuracy of Bedrock KB is insufficient. The risk of returning the wrong state's policy or missing a critical compliance rule is too high.

Q10: What are Docker networking concepts? How do LoanIQ services communicate internally?

A: Docker Compose creates a default network named {project_name}_default. All services in the Compose file join this network automatically.

Service discovery by name: Inside the network, services are DNS-addressable by their service name: - API container connects to PostgreSQL: postgresql://postgres:5432/loaniq_db (hostname = postgres) - If ollama service was added, API would connect: http://ollama:11434/v1

This is Docker's internal DNS — only works inside the Docker network, not from the host.

Port mapping (ports: "8000:8000"): Exposes the container's port 8000 to the host's port 8000. From your browser: http://localhost:8000 → API container. Without port mapping, the API is only accessible from inside the Docker network.

Bridge network (default): Isolated from the host network. Services can reach each other by name, but the host needs explicit port mappings to reach services. Good for security.

Host network mode (network_mode: host): Container shares the host's network stack. No isolation — container can access all host ports. Slightly faster (no NAT) but rarely used in production. PostgreSQL should NEVER use host mode in production.

In LoanIQ production: An Application Load Balancer sits in front of ECS. The ALB's target group hits http://ecs-task-ip:8000. PostgreSQL runs in a private subnet, not reachable from internet — only from the ECS task's security group.
