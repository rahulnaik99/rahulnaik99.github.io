RAG — Retrieval Augmented Generation

Module 1: Ingestion Pipeline

What is RAG? RAG is a technique where, instead of relying on what an LLM "memorized" during training, you first retrieve relevant documents from a database, then feed those documents into the LLM as context so it can generate a grounded, accurate answer.

Think of it like an open-book exam: the LLM doesn't need to remember everything — it just reads the relevant pages you hand it.

Sub-topic 1: Document Loading

What, Why, How

What: Document loading is the first step of the ingestion pipeline. It reads raw files (PDFs, Word docs, HTML, CSV) and extracts plain text from them.

Why: LLMs cannot read binary file formats like PDFs. You must convert them to plain text first. The quality of your extracted text directly impacts retrieval quality — garbage in, garbage out.

How: Libraries like PyMuPDF (fitz), pypdf, or pdfplumber open the file page by page and extract text. For scanned PDFs with no embedded text, OCR (Optical Character Recognition) tools like Tesseract or AWS Textract are required.

Alternatives & Why We Used PyMuPDF

  • pypdf: Pure Python, no C dependency, but struggles with complex layouts. We chose PyMuPDF because it handles real-world mortgage PDFs (multi-column, tables, headers) far better.
  • pdfplumber: Best for tables but slower; we don't need table extraction in the policy ingestion path.

  • LangChain PyPDFLoader: Wraps pypdf, convenient but less control over per-page extraction.

We chose PyMuPDF because it's the fastest high-accuracy library for digital PDFs, and we need per-page extraction for citation metadata.

Sub-topic 2: Text Cleaning

What, Why, How

What: After extraction, raw PDF text is messy. It contains headers, footers, page numbers, hyphenated line breaks (mort-\ngage), and random whitespace. Text cleaning normalizes all of this.

Why: Dirty text produces bad embeddings. If chunk 1 ends with mort- and chunk 2 starts with gage, the word "mortgage" appears nowhere in your vector database — searches for "mortgage" will miss it.

How: A series of regex replacements and string operations: 1. Remove repeated headers/footers using pattern matching 2. Fix hyphenated line breaks (re.sub(r'-\n', '', text)) 3. Normalize whitespace 4. Handle Unicode normalization

In LoanIQ Project

In pdf_ingestion.py, basic cleaning is applied after extraction:

text = " ".join(text.split())  # Collapse all whitespace including \n into single spaces

Implementation

import re
import unicodedata

def clean_pdf_text(text: str) -> str:
    """Full text cleaning pipeline for PDF-extracted text."""
    # Fix hyphenated line breaks (common in justified PDF text)
    text = re.sub(r'(\w)-\n(\w)', r'\1\2', text)

    # Remove page numbers (standalone numbers on a line)
    text = re.sub(r'\n\s*\d+\s*\n', '\n', text)

    # Remove common header/footer patterns
    text = re.sub(r'CONFIDENTIAL.*?\n', '', text, flags=re.IGNORECASE)

    # Normalize unicode (fix smart quotes, em-dashes, etc.)
    text = unicodedata.normalize('NFKC', text)

    # Collapse multiple newlines to max two (preserves paragraphs)
    text = re.sub(r'\n{3,}', '\n\n', text)

    # Normalize whitespace within lines
    text = re.sub(r'[ \t]+', ' ', text)

    return text.strip()

Alternatives

  • spaCy sentence splitter: More accurate but much heavier for pure cleaning
  • NLTK: Overkill for text normalization
  • Hand-tuned regexes: What LoanIQ uses — simple, fast, and predictable for known document formats

Sub-topic 3: Chunking Strategies

What, Why, How

What: Chunking splits a long document into smaller pieces (chunks). Each chunk gets its own embedding vector and is stored separately in the vector database.

Why: You cannot embed an entire 200-page PDF as one vector — you'd lose all specificity. Also, LLMs have context windows. You need small, retrievable pieces. But chunks too small lose context.

How: Split on natural boundaries (paragraphs, sentences) with configurable size and overlap.

Alternatives

  • TokenTextSplitter: Splits by actual token count — more precise for LLM context budget but slower
  • SentenceTransformersTokenTextSplitter: Splits by the embedding model's token count — ensures no chunk overflows the embedding model's context
  • Semantic Chunking: Uses embedding similarity shifts to find topic boundaries; expensive but produces the most meaningful chunks

Why Recursive Character Splitting? It's the best balance of quality and speed. It produces coherent chunks that respect document structure without the computational cost of semantic splitting.

Sub-topic 4: Metadata Preservation

What, Why, How

What: Alongside the chunk text, you store structured metadata: page number, section title, document name, character offsets, etc.

Why: Without metadata, you can't cite sources. "The answer is X" is useless in production — you need "The answer is X (Source: Policy 3, Section 4.2, Page 18)". Also, metadata enables pre-filtering during retrieval (e.g., "only search chunks from state=CA").

How: Extract metadata during ingestion, store it in separate columns (not mixed into the chunk text), and attach it to every retrieved chunk.

import re

def detect_section_header(text: str) -> str | None:
    """Detect if text starts with a section header using regex."""
    patterns = [
        r'^(\d+\.\d+\.?\s+[A-Z][A-Za-z\s]{5,50})$',   # "4.2 Credit Score Requirements"
        r'^(Section\s+\d+[:\s]+.{5,50})$',              # "Section 4: Eligibility"
        r'^([A-Z][A-Z\s]{5,40})$',                      # "CREDIT REQUIREMENTS"
    ]
    first_line = text.split('\n')[0].strip()
    for pattern in patterns:
        match = re.match(pattern, first_line, re.MULTILINE)
        if match:
            return match.group(1).strip()
    return None

def build_chunk_metadata(
    chunk_text: str,
    page_number: int,
    doc_title: str,
    chunk_index: int
) -> dict:
    return {
        "page_number": page_number,
        "section_title": detect_section_header(chunk_text),
        "document_title": doc_title,
        "chunk_index": chunk_index,
        "char_count": len(chunk_text),
        "token_estimate": len(chunk_text) // 4,  # 4 chars ≈ 1 token
    }

Sub-topic 5: Embedding Models

What, Why, How

What: An embedding model converts text into a dense vector of floating-point numbers (e.g., 1536 floats). Semantically similar texts produce vectors that are close together in this high-dimensional space.

Why: You cannot do "semantic search" on raw text — computers can't compare meaning directly. Vectors enable cosine similarity calculations that measure semantic closeness.

How: Pass text to an embedding API or local model, receive a list of floats. These floats capture the "meaning" of the text.

Sub-topic 6: Vector Stores

What, Why, How

What: A vector store (vector database) stores embedding vectors alongside their metadata and supports fast approximate nearest neighbor (ANN) search — finding the K most similar vectors to a query vector.

Why: Regular SQL = comparisons don't work on vectors. You need specialized indexes (like HNSW) that can efficiently find vectors with high cosine similarity.

How: Store (id, embedding_vector, metadata). At query time, compute cosine similarity between the query vector and all stored vectors, return the top-K results.

HNSW Index — How It Navigates
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Layer 2 (coarse): [A] ─────────────────── [Z]
Layer 1 (medium): [A] ──── [M] ──────── [Z]  
Layer 0 (fine):   [A][B][C][D]...[M][N]...[Z]

Query → Enter at top layer → Navigate toward query →
Drop to finer layer → Repeat → Final neighbors found
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The SQL query uses <=> (cosine distance operator):

SELECT *, 1 - (pc.embedding <=> CAST(:embedding AS vector)) AS similarity_score
FROM policy_chunks pc
WHERE 1 - (pc.embedding <=> ...) >= 0.7   -- similarity threshold
ORDER BY pc.embedding <=> CAST(:embedding AS vector)
LIMIT 10

<=> = cosine distance (0 = identical, 2 = opposite). Similarity = 1 - distance.

Why pgvector?

Why not Pinecone? pgvector means one database for everything — application data, borrower records, and vectors. No extra infrastructure, no syncing, no extra cost. For LoanIQ's scale, pgvector is more than sufficient.

Sub-topic 7: Idempotency & Versioning

What, Why, How

What: Idempotency means running the ingestion pipeline multiple times on the same document produces the same result without creating duplicate records.

Why: Policy PDFs get updated. You need to re-ingest them without flooding the DB with duplicates. Also, if the pipeline crashes halfway, you need safe restarts.

How: Hash the document content, check if that hash already exists in the DB before ingesting. On re-ingestion, soft-delete old chunks and insert new ones.

In LoanIQ Project

import hashlib

def compute_document_hash(file_path: str) -> str:
    """SHA-256 hash of file contents — changes if file changes."""
    sha256 = hashlib.sha256()
    with open(file_path, "rb") as f:
        for chunk in iter(lambda: f.read(8192), b""):
            sha256.update(chunk)
    return sha256.hexdigest()

async def ingest_with_idempotency(file_path: str, session):
    doc_hash = compute_document_hash(file_path)

    # Check if already ingested with same content
    existing = await session.execute(
        select(PolicyDocument).where(PolicyDocument.content_hash == doc_hash)
    )
    if existing.scalar_one_or_none():
        return "already_exists"

    # Soft-delete old version if file was updated
    old = await session.execute(
        select(PolicyDocument).where(PolicyDocument.filename == Path(file_path).name)
    )

10 Interview Questions — RAG Ingestion Pipeline

Q1: What's the difference between pypdf, pdfplumber, and PyMuPDF? Which would you choose for production?

A: All three extract text from PDFs, but differ in accuracy, speed, and table handling. - pypdf: Pure Python, no C dependency, easiest to install. Works for simple PDFs but struggles with complex layouts, multi-column text, and tables. - pdfplumber: Best table detection — identifies cell boundaries, row/column structure. Slower due to detailed parsing. - PyMuPDF (fitz): Fastest, C-based. Excellent for complex layouts and digital PDFs. Cannot do OCR.

In LoanIQ, we use PyMuPDF for its speed and accuracy on digital mortgage policy PDFs. We'd add Textract only if we encountered scanned documents.

Q2: Why do we add page numbers to chunk text during extraction? Isn't that metadata, not content?

A: We tag the text with [PAGE N] markers and then store page numbers in the metadata separately. The page tag in the text helps the chunker know where page breaks occur, which is important for two reasons: 1. Citation accuracy: When we retrieve chunk X, we can tell the user "Source: Page 18" — critical for compliance in mortgage. 2. Boundary awareness: Prevents chunks from combining content that spans unrelated pages.

In LoanIQ: f"[PAGE {i+1}]\n{text}" is prepended before splitting so that even if a chunk spans a page boundary, we can still trace it.

Q3: Explain the RecursiveCharacterTextSplitter. Why is it called "recursive"?

A: It's "recursive" because it tries a list of separators in priority order and recursively falls back to the next one if chunks are still too large.

Priority: ["\n\n", "\n", ". ", " ", ""]

  1. First tries to split on \n\n (paragraph breaks) — best quality
  2. If any chunk is still too big, tries \n (line breaks)
  3. Still too big? Tries . (sentence ends)
  4. Still too big? Splits on spaces
  5. Last resort: splits mid-character

This ensures we cut at the most natural boundary possible. In LoanIQ, mortgage policy paragraphs rarely exceed 1000 chars, so most splits happen at \n\n level.

Q4: What chunk size and overlap did you use in LoanIQ and why?

A: chunk_size=1000 characters, chunk_overlap=200 characters.

  • 1000 chars (~250 tokens) is enough for a complete policy rule with context, but small enough that retrieved chunks are specific, not overly broad.
  • 200 char overlap (~50 tokens, ~20% of chunk) prevents information loss at boundaries. A sentence like "The maximum LTV for FHA loans is..." won't be split such that "maximum LTV for FHA loans is" is in one chunk and "...80%" is in the next — the overlap ensures both chunks contain the complete rule.

For production, we tune these based on RAGAS context precision/recall scores.

Q5: What is a vector embedding? Explain it to a 5-year-old and then explain it technically.

A:

Simple explanation: Imagine placing every sentence in a giant room. Sentences with similar meanings stand close to each other. "The DTI must be below 43%" stands near "Debt-to-income ratio cannot exceed 43%." An embedding is the (x, y, z...) address of a sentence in that room.

Technical explanation: An embedding model (like text-embedding-3-small) is a neural network trained to compress text into a dense vector of floats (e.g., 1536 floats). The training objective ensures that semantically similar texts produce vectors with high cosine similarity. The model captures meaning, not keywords: "DTI limit" and "maximum debt ratio" will have high cosine similarity even though they share no words.

Formula: cosine_similarity(A, B) = (A · B) / (|A| × |B|) — ranges from -1 to 1, where 1 = identical direction = same meaning.

Q6: Why use pgvector instead of Pinecone for LoanIQ?

A: Several reasons:

  1. Single database: LoanIQ already uses PostgreSQL for borrower records, loan applications, and decisions. pgvector adds vector capabilities to the same DB — no separate service, no data sync, no extra infrastructure cost.
  2. SQL joins: We can join policy_chunks (vector) with policy_documents (metadata) and loan_applications (application data) in a single SQL query. Pinecone can't do that.
  3. Scale: LoanIQ has ~500-1000 policy chunks. pgvector with HNSW index handles millions of vectors easily — no need for a dedicated vector DB at this scale.
  4. Privacy: Data stays in our PostgreSQL instance. Pinecone is a cloud service — a concern for financial data.

Pinecone would make sense at 100M+ vectors or when multiple services need to share a vector store.

Q7: What is HNSW and how does it make vector search fast?

A: HNSW (Hierarchical Navigable Small World) is a graph-based approximate nearest neighbor index.

Without HNSW, finding the nearest vector to a query requires comparing with every stored vector — O(n) time. With 1M vectors, that's 1M cosine calculations per query.

HNSW builds a layered graph: - Top layers: sparse, long-range connections — navigation highways - Bottom layer: dense, short-range connections — fine-grained search

At query time: enter at top layer → greedily navigate toward the query vector → drop to lower layer → repeat until bottom → you've found approximate nearest neighbors.

Result: O(log n) search time. In pgvector: CREATE INDEX ON policy_chunks USING hnsw (embedding vector_cosine_ops);

The tradeoff: HNSW finds approximate neighbors (may miss the absolute nearest), but in practice it's 99%+ accurate while being 100x faster than exact search.

Q8: What's the difference between cosine similarity, dot product, and L2 distance for vector search?

A:

  • Cosine similarity: Measures the angle between two vectors, ignoring magnitude. sim = (A·B)/(|A||B|). Best for text embeddings where the length of the vector shouldn't affect similarity. Used in LoanIQ.
  • Dot product: A·B — measures both direction AND magnitude. Use when embedding magnitude carries meaning (e.g., some models are trained with dot product objective).
  • L2 distance (Euclidean): Physical distance in vector space. Less appropriate for text because two embeddings could be far apart in L2 but have the same direction.

Rule of thumb: For normalized embeddings (unit vectors), cosine similarity = dot product (they're mathematically equivalent). Most text embedding models produce approximately normalized vectors, so either works.

In pgvector: <=> = cosine distance, <#> = negative dot product, <-> = L2 distance.

Q9: How do you handle duplicate ingestion? What happens if the same PDF is ingested twice?

A: Idempotency is built into the ingestion service:

  1. Hash check: Compute SHA-256 of the file. If the same hash already exists in policy_documents, skip ingestion.
  2. Filename + soft-delete: If the filename exists but hash differs (file was updated), soft-delete the old document (is_active = False) and its chunks before inserting the new ones. The old data is preserved for audit trail.
  3. Atomic transaction: The entire ingestion (document + all chunks) happens in one database transaction. If it fails halfway, everything is rolled back — no half-ingested documents.

This means you can run python ingest_policies.py safely in a CI/CD pipeline on every deployment.

Q10: What is semantic chunking and when would you use it instead of RecursiveCharacterTextSplitter?

A: Semantic chunking embeds the text first, then splits where the embedding similarity drops significantly between consecutive sentences. It detects topic changes rather than splitting at character counts.

How it works: 1. Split document into sentences 2. Embed each sentence 3. Compute cosine similarity between adjacent sentences 4. Split where similarity drops below a threshold (topic change)

When to use it: - Documents with clearly defined topic sections that don't align with paragraph breaks (e.g., transcripts, mixed-topic articles) - When retrieval precision is more important than ingestion speed

Why LoanIQ doesn't use it: - 3-5x slower (requires embedding every sentence during ingestion) - Policy PDFs have clear paragraph structure — RecursiveCharacterTextSplitter already produces coherent policy-rule chunks - Semantic chunking shines when topic boundaries are unpredictable; mortgage policies are well-structured documents

Cost of semantic chunking: ~5x more embedding API calls during ingestion.
