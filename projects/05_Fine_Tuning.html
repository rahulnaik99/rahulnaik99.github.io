Fine-Tuning: LoRA & QLoRA

Complete Study Guide

What is Fine-Tuning? Fine-tuning adapts a pre-trained LLM to a specific task or domain by continuing to train it on domain-specific examples. Instead of re-training from scratch (impossibly expensive), you adjust the model using your data.

LoanIQ fine-tunes Llama 3.1 8B to generate mortgage ratio analysis narratives in the exact format that underwriters expect — something GPT-4o doesn't do out-of-the-box.

Sub-topic 1: Why Fine-Tune?

What, Why, How

What: Fine-tuning is the process of taking a pre-trained foundation model and continuing training it on domain-specific examples so it learns specialized knowledge or output style.

Why fine-tune vs. prompt engineering?

Approach Cost per Call Quality Privacy Latency
GPT-4o with detailed prompts ~$0.015/1K tokens High Data sent to OpenAI ~2-3s
Fine-tuned Llama 3.1 8B local $0 (inference) Higher for domain task Fully private ~0.5s
GPT-4o fine-tuned ~$0.005/1K tokens High Data sent to OpenAI ~2-3s

LoanIQ rationale for fine-tuning:

  1. Domain-specific language: Mortgage terms like "PITI", "HLTV", "conforming limit", "Freddie Mac overlay" appear rarely in GPT training data. A fine-tuned model speaks fluent mortgage language.

  2. Output style control: Underwriters need narratives in a specific format: "DTI: 42.5% (Back-End). Analysis: Borrower's debt ratio exceeds the standard 43% threshold; however, compensating factors include..." — hard to achieve consistently with prompting alone.

  3. Cost at scale: 500 loan decisions/day × average 3000 tokens = 1.5M tokens/day. At GPT-4o pricing ($15/1M tokens) = $22.50/day = $8,200/year. Fine-tuned local model: $0/call.

  4. Privacy: Borrower PII and financial data never leaves your infrastructure. No GDPR risk, no data sharing agreements.

Sub-topic 2: Full Fine-Tuning vs LoRA

The Problem with Full Fine-Tuning

Full fine-tuning updates all model weights. Llama 3.1 8B has 8 billion parameters. Each parameter stored as float16 = 2 bytes → 16GB just to store the model. Training requires 3× the storage for optimizer states → 48GB+. Consumer GPUs top out at 24GB. Impractical.

LoRA — Low-Rank Adaptation

Key insight: Weight updates during fine-tuning are low-rank. You don't need to update all 8B parameters — just add small "adapter" matrices that capture the domain-specific changes.

LoRA Architecture:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Frozen base weights: W (d × d matrix)
LoRA adapters: A (d × r) and B (r × d), where r << d

Forward pass:
  h = W·x + (lora_alpha/r) × B·A·x

W is frozen (never updated). Only A and B are trained.

Example:
  W: 4096 × 4096 = 16,777,216 parameters (frozen)
  A: 4096 × 16  =    65,536 parameters (trainable)
  B: 16  × 4096 =    65,536 parameters (trainable)

Trainable params = 131,072 vs 16,777,216 = 0.78% of one layer
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Why this works: The hypothesis (validated empirically) is that weight updates during fine-tuning can be expressed as a low-rank matrix. If r=16 captures the needed update, you only train 0.1% of parameters.

Key LoRA Hyperparameters

Parameter What it controls LoanIQ value Notes
r (rank) Size of A and B matrices 16 Higher = more capacity, more VRAM
lora_alpha Scaling: scale = alpha/r 16 Keep equal to r for effective scale=1
target_modules Which weight matrices to adapt q,k,v,o,gate,up,down proj All attention + FFN layers
lora_dropout Dropout on adapter 0 0 is optimal with QLoRA (Dettmers 2023)

Sub-topic 3: QLoRA — Quantized LoRA

What, Why, How

What: QLoRA (Quantized LoRA) extends LoRA by quantizing the frozen base model to 4-bit precision. The LoRA adapters are still trained in 16-bit.

Why: Quantizing 8B float16 model to 4-bit reduces VRAM from 16GB to 4GB — now fits on a 16GB consumer GPU (RTX 3080, A10G, Colab A100).

QLoRA Memory Calculation:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Llama 3.1 8B model:

Full precision (FP32):   8B × 4 bytes = 32 GB  ← requires 2× A100
Half precision (FP16):   8B × 2 bytes = 16 GB  ← requires A100 (80GB for training)
4-bit quantized (NF4):   8B × 0.5 bytes = 4 GB ← fits on 16GB GPU ✅

LoRA adapters (FP16): ~50MB for r=16
Optimizer states (FP16): ~200MB
Activations: ~2GB

Total QLoRA training VRAM: ~6-8GB ← fits on RTX 3080!
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

NF4 quantization: Normal Float 4-bit is designed specifically for neural network weights. Neural network weights follow a normal (Gaussian) distribution. NF4 bins are spaced to optimally represent a normal distribution — not equally spaced like standard INT4. This preserves more information.

Double quantization: Quantize the quantization constants too (the scale factors used to dequantize). Saves ~0.37 bits/parameter — ~3GB savings for 70B models. Less significant for 8B.

In LoanIQ Project

From scripts/finetune_ratio_model.py:

from unsloth import FastLanguageModel
import torch

model, tokenizer = FastLanguageModel.from_pretrained(
    model_name="unsloth/Meta-Llama-3.1-8B",
    max_seq_length=512,
    dtype=None,        # auto-detect bfloat16 on A100
    load_in_4bit=True, # QLoRA — quantize to NF4
)

model = FastLanguageModel.get_peft_model(
    model,
    r=16,              # LoRA rank
    target_modules=[
        "q_proj", "k_proj", "v_proj", "o_proj",    # attention
        "gate_proj", "up_proj", "down_proj",         # FFN
    ],
    lora_alpha=16,
    lora_dropout=0,    # 0 is optimal with QLoRA
    use_gradient_checkpointing="unsloth",  # 30% less VRAM
)

Why Unsloth? Unsloth implements fused kernels for LoRA training — 2× faster training speed with 60% less VRAM vs standard HuggingFace PEFT. For a 45-minute Colab job, this cuts training to ~20 minutes.

Sub-topic 4: Training Setup

SFTTrainer Configuration

from trl import SFTTrainer
from transformers import TrainingArguments

trainer = SFTTrainer(
    model=model,
    tokenizer=tokenizer,
    train_dataset=dataset,
    dataset_text_field="text",
    max_seq_length=512,
    args=TrainingArguments(
        per_device_train_batch_size=2,
        gradient_accumulation_steps=4,   # effective batch = 2×4 = 8
        num_train_epochs=3,
        learning_rate=2e-4,              # standard LoRA LR
        bf16=True,                       # bfloat16 on A100
        logging_steps=10,
        optim="adamw_8bit",              # 8-bit Adam saves VRAM
        output_dir="./checkpoints",
    ),
)
trainer.train()

Gradient Accumulation

Problem: With batch size 2 on a 16GB GPU, gradients are noisy (high variance). Ideal batch size is 8-16.

Solution: gradient_accumulation_steps=4 means: 1. Forward + backward pass on batch of 2 → accumulate gradients 2. Repeat 4 times → total gradient from 8 samples 3. Update weights once

Effective batch size = per_device_batch_size × gradient_accumulation_steps = 2 × 4 = 8

No extra VRAM needed. Computation time = 4× more steps, but same quality as batch 8.

Why 500 examples? Rule of thumb for meaningful domain adaptation via LoRA: - <100: Likely overfits, doesn't generalize - 200-500: Sweet spot for style/format learning - 1000+: Needed for deep knowledge injection

Edge Cases

Include borderline scenarios deliberately: - DTI exactly 45% (right at limit) - LTV exactly 80% (PMI threshold) - Credit 619 (one point below minimum) - Combinations: Good credit + bad DTI, etc.

These edge cases ensure the model handles real-world ambiguity, not just easy clear-cut cases.

Sub-topic 6: Evaluation

Training Loss Target

A decreasing training loss is necessary but not sufficient: - Target: loss < 0.5 after 3 epochs - Steady decrease without sudden spikes (spikes indicate bad training examples) - If loss plateaus early, may need more epochs or higher learning rate

Catastrophic Forgetting

Problem: Fine-tuning on mortgage data can cause the model to forget general capabilities (writing, reasoning, coding). Called "catastrophic forgetting."

Mitigation: 1. LoRA protects against it — base weights are frozen; only adapters change. The base model's general capabilities are preserved. 2. Low learning rate (2e-4): Gentle updates don't overwrite too much 3. Evaluation on general tasks before and after fine-tuning

With QLoRA, catastrophic forgetting is minimal because the 8B base model is 4-bit frozen — its general knowledge is structurally preserved.

10 Interview Questions — Fine-Tuning

Q1: Why did LoanIQ fine-tune a model instead of just improving the system prompt?

A: Four reasons:

  1. Output format consistency: Underwriters need narratives in a very specific structured format — numbered sections, specific headers, quantitative thresholds cited correctly. Even with detailed few-shot examples, GPT-4o produces inconsistent formatting across different loan profiles. The fine-tuned model was trained on 500 examples in the exact format, so it reliably produces correct structure.

  2. Domain terminology: Mortgage language is specialized — "PITI", "HLTV", "Freddie Mac overlay", "qualifying income". The fine-tuned model uses this terminology naturally.

  3. Cost: The RatioCalculationAgent runs on every loan decision. At scale, using GPT-4o for this agent alone would cost thousands per month. Fine-tuned Llama 3.1 8B runs locally in Ollama at $0/call.

  4. Latency: Local Ollama inference: ~300ms. GPT-4o API: ~2000ms. For a 7-agent pipeline, saving 1.7 seconds per agent multiplies significantly.

Prompt engineering reaches diminishing returns for format-critical, high-volume tasks. Fine-tuning is the right tool here.

Q2: Explain LoRA. What are the A and B matrices and why does this reduce training cost?

A: LoRA (Low-Rank Adaptation) hypothesizes that weight updates during fine-tuning are "low-rank" — they can be expressed as the product of two small matrices.

Instead of updating W (a 4096×4096 matrix = 16M parameters), LoRA learns: - A: 4096×16 matrix (65,536 parameters) - B: 16×4096 matrix (65,536 parameters)

Forward pass: h = (W + α/r × B·A) · x

W is frozen. Only A and B are updated. In a full model, we add LoRA to every attention projection and FFN layer. Total trainable parameters ≈ 0.1% of the model.

Why 0.1% is enough: Domain adaptation doesn't require re-learning how to reason or use language — just learning the specific output style and domain terminology. A small delta on frozen weights is sufficient for this.

Training A and B matrices requires proportionally less GPU memory, less gradient computation, and the resulting "adapter" file is only ~50MB vs 16GB for the full model.

Q3: What is the difference between LoRA and QLoRA?

A: LoRA freezes the base model weights but keeps them in full precision (FP16 = 16-bit). To load and fine-tune Llama 3.1 8B in FP16, you need 16GB VRAM just for the model parameters, plus optimizer states.

QLoRA adds: quantize the base model to 4-bit (NF4) before training. This reduces base model VRAM from 16GB to ~4GB. The LoRA adapters are still trained in 16-bit (you can't effectively train in 4-bit — gradients become too noisy).

LoRA:   FP16 base (16GB) + FP16 adapters (~50MB)
QLoRA:  NF4 base  (4GB)  + FP16 adapters (~50MB)

The key insight: NF4 is a special 4-bit format designed for normally distributed weights (which neural network weights are). During the forward pass, NF4 values are dequantized to FP16 for computation, then gradients flow back to the FP16 adapters.

LoanIQ uses QLoRA to train on Google Colab's A100 (~16GB VRAM) — LoRA alone wouldn't fit.

Q4: What is gradient_accumulation_steps and why is it needed?

A: gradient_accumulation_steps is a technique to simulate a larger batch size when GPU memory is limited.

With batch_size=2, each forward+backward pass uses only 2 samples. The gradient estimate is noisy. You'd ideally use batch_size=16 but that requires 8× more activation memory.

With gradient_accumulation_steps=4: 1. Forward+backward on 2 samples → store gradients (don't update weights) 2. Forward+backward on next 2 samples → ADD to stored gradients 3. Forward+backward on next 2 samples → ADD 4. Forward+backward on next 2 samples → ADD → NOW update weights

Effective batch size = 2×4=8. Memory usage is still for 2 samples at a time, but the gradient quality is as if you used batch 8.

The only downside: 4× more forward/backward passes per weight update = slightly slower, but no quality loss compared to true batch 8.

Q5: What are target_modules in LoRA configuration? Why do we add adapters to so many layers?

A: target_modules specifies which weight matrices in the model get LoRA adapters. In Llama 3.1:

  • q_proj, k_proj, v_proj: Query, Key, Value projection matrices in attention layers. These control how each token attends to others — critical for reasoning and language understanding.
  • o_proj: Output projection of attention — combines multi-head attention outputs
  • gate_proj, up_proj, down_proj: The three weight matrices in the SwiGLU FFN layer — control how token representations are transformed

LoanIQ adds adapters to all 7 module types. Why all of them? More target modules = more trainable parameters = more capacity to learn the domain. For a specific output style task (mortgage narratives), adapting both attention and FFN gives the model enough capacity to learn the exact narrative format.

You could reduce target modules to just q_proj and v_proj (as in the original LoRA paper) to save VRAM, but LoanIQ prioritizes quality over marginal VRAM savings.

Q6: What is the teacher-student approach to training data generation? Why use GPT-4o as teacher?

A: Teacher-student (aka knowledge distillation) trains a smaller, cheaper model to replicate the outputs of a larger, more capable model.

In LoanIQ: 1. Teacher (GPT-4o): Given loan metrics → generates high-quality underwriter narratives. Cost: ~$30 for 500 examples (one-time). 2. Student (Llama 3.1 8B): Trained on GPT-4o's outputs → learns to generate similar narratives. Cost: ~$10/month on Colab Pro (training). $0/inference (local Ollama).

Why GPT-4o as teacher: - GPT-4o produces the highest-quality structured mortgage narratives - Its outputs become the "gold standard" that Llama learns to match - After fine-tuning, Llama 3.1 8B matches GPT-4o quality on this specific task, at 1/10 the inference cost

This is the key economic argument: spend $30 once on GPT-4o for training data → save thousands per month on inference.

Q7: How do you evaluate a fine-tuned model? What's "LLM-as-judge"?

A: Three evaluation methods in LoanIQ:

  1. Training loss: Should decrease steadily, target < 0.5. Necessary but not sufficient — loss just measures how well the model memorized training data.

  2. LLM-as-judge: Use GPT-4o-mini (a strong but cheap model) to score fine-tuned Llama's outputs against reference answers on a held-out test set:

judge_prompt = """
Score 0-10: Does the model's narrative correctly identify:
- LTV violation (if applicable)?
- DTI guideline compliance?
- Professional underwriter language?
Model output: {model_output}
Reference: {reference}
Score:
"""
  1. Human evaluation: Show real underwriters 20 outputs from GPT-4o and 20 from fine-tuned Llama (blind). Ask which is better. If they prefer fine-tuned outputs ≥ 80% of the time, the model is production-ready.

LLM-as-judge advantage: No labeled test set needed. GPT-4o-mini can assess "does this narrative correctly identify the DTI violation?" without you providing explicit labels.

Q8: What is catastrophic forgetting? How does QLoRA mitigate it?

A: Catastrophic forgetting is when fine-tuning a model on domain-specific data causes it to lose its general capabilities. Example: After fine-tuning on mortgage data, the model can't write Python code anymore, or its reasoning ability degrades.

In standard fine-tuning, all weights are updated — general knowledge encoded in weights gets overwritten by mortgage knowledge.

QLoRA mitigates this in two ways: 1. Frozen base model: The 8B base model weights are frozen (never updated). All general knowledge (language understanding, reasoning, coding) is structurally preserved. 2. Small adapters only: Only the tiny A and B LoRA matrices are updated. These are additive to the frozen weights — they layer domain knowledge on top without destroying the foundation.

Result: After QLoRA fine-tuning, Llama 3.1 8B can still write code, answer general knowledge questions, AND generate mortgage narratives. The base stays intact; the adapters add the specialization.

Q9: Explain NF4 quantization. Why is it better than simple INT4?

A: INT4 quantization equally spaces 16 values between min and max weight values. But neural network weights follow a bell curve (normal distribution) — most values cluster near zero, few values are extreme.

INT4: [-1.0, -0.87, -0.73, ..., 0.73, 0.87, 1.0] — equally spaced, wastes many bins on rare extreme values

NF4 (Normal Float 4): Bins are spaced according to quantiles of a standard normal distribution: [-1.0, -0.69, -0.53, ..., 0.00, ..., 0.53, 0.69, 1.0] — denser bins near zero where most weights cluster

This means more quantization levels are used for the values that actually appear most frequently → less quantization error → better model quality at the same 4-bit budget.

NF4 was introduced in the QLoRA paper (Dettmers et al., 2023) and is the default in HuggingFace bitsandbytes.
