Why Fine-Tune
  • Domain-specific language — mortgage terms, medical, legalMUST
  • Output style control — underwriter narrative formatMUST
  • Cost reduction — fine-tuned 8B > GPT-4o quality at $0 per callMUST
  • Latency — local model, no API round tripGOOD
  • Privacy — no data sent to OpenAIGOOD
Full Fine-Tuning vs LoRA
  • Full fine-tuning — updates all weights, needs 80GB+ GPU, impracticalMUST
  • LoRA — freeze base weights, add small A×B adapter matricesMUST
  • Output = W·x + scale × A·B·x — tiny delta on frozen weightsMUST
  • Only 0.1% of params trained — fits on consumer GPUMUST
QLoRA Specifics
  • Q = quantize base model to 4-bit — 8B model goes from 16GB to 4GBMUST
  • LoRA adapters trained in 16-bit on top of 4-bit baseMUST
  • NF4 quantization — normal float 4, optimal for normally distributed weightsGOOD
  • Double quantization — quantize the quantization constants tooNICE
Key Hyperparameters
  • r (rank) — size of A,B matrices. r=16 sweet spot for 8BMUST
  • lora_alpha — scale = alpha/r. Keep equal to rMUST
  • target_modules — q_proj, k_proj, v_proj, o_proj, gate/up/down_projMUST
  • lora_dropout — 0 is optimal with QLoRAGOOD
  • learning_rate — 2e-4 standard for LoRAGOOD
Training Setup
  • Unsloth — optimized for consumer GPUs, 2x faster than HuggingFaceMUST
  • SFTTrainer from TRL — supervised fine-tuningMUST
  • Alpaca prompt format — instruction / input / outputMUST
  • gradient_accumulation_steps — simulate larger batch on small GPUMUST
